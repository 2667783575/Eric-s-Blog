关于cpp中const返回值

背景 两个月前看了一点Effective Modern Cpp，后来听说作者Scott Mayer写了三本书，Cpp三部曲(E C, More E C, E Modern C)。于是搁置了Effective Modern Cpp，转而去看Effective Cpp。当时看到作者的一个观点是函数返回值优先const 即优先写 1const std::string func()... 而不是 1std::string func()... 原因是有人会在if语句里做判断，发生typo，将判断写成赋值 如 1Matrix64 a, b, c; 2... 3if (a * b = c){ // should be "==" 4 5} 若*的重载函数返回const Matrix64，编译器就会报错，否则就难以找到错误发生的地方。 听起来非常有道理，于是书上说 一个函数返回一个 constant value（常量值），常常可以在不放弃安全和效率的前提下尽可能减少客户的错误造成的影响。 然而今天机缘巧合之下我去问dicksip我们是不是应该在cpp的函数返回值上默认const，它却给出了否定的答案。 其实原因也非常简单，只是我之前没有去想 为什么我们不应该广泛使用const返回值? 原因1 降低性能(阻止移动语义,RVO,NRVO) 在现代cpp中，函数返回值const会影响移动语义，移动语义需要非常量右值 在没有const的情况下 1Data getData(){ 2 Data data; 3 ... 4 return data; 5} 6Data receiver = getData();// 触发移动语义，甚至RVO,NRVO 7 8// 如果 9// const Data getData(); 10// 则只能触发拷贝。 原因2 阻碍现代范式 如 链式调用(Fluent Interface) 1class MyString{ 2public: 3 MyString& toUpper(); 4} 5 6 7proccessString.toUpper();//没有const时才能写 就地操作 1void modifyValue(std::string&& str); 2 3 4modifyValue(getValue()) //高效接受右值处理 一种惯用法 1// 从流中读取数据并检查是否成功 2if (std::cin >> value) { ... } 3 4// 在循环中获取锁并检查是否成功 5while (std::unique_lock lock(mutex, std::try_to_lock)) { ... } 6 7// 获取一个资源指针并检查是否有效 8if (auto ptr = acquireResource()) { ... } 这里unique_ptr和unique_lock等都必须接受非常量右值，如果const就无法构造。 ...
