提供一个简明的 API 来逐步构建一个复杂对象，将其构建过程与它的表示分离开。#

引入#

先来看一段C#自带的Builder模式的应用

StringBuilder

假设我们想要构建Html元素

 1static void Main(string[] args)
 2{
 3    var hello = "hello";
 4    var sb = new StringBuilder();
 5    sb.Append("<p>");
 6    sb.Append(hello);
 7    sb.Append("</p>");
 8    Console.WriteLine(sb.ToString());
 9    sb.Clear();
10    var words = new[] { "hello", "world" };
11    sb.Append("<ul>");
12    foreach (var word in words)
13    {
14        sb.AppendFormat("<li>{0}<li>", word);
15    }
16    sb.Append("</ul>"); 
17    Console.WriteLine(sb.ToString());
18}

我们现在想要写一个HtmlElement类将Html元素封装起来，它需要能够包含标签名，文本和子元素。

 1public class HtmlElement
 2{
 3    public string Name, Text;
 4    public List<HtmlElement> Elements = []; 
 5
 6    public HtmlElement()
 7    {
 8        
 9    }
10
11    public HtmlElement(string name, string text)
12    {
13        Name = name ?? throw new ArgumentNullException(nameof(name));
14        Text = text ?? throw new ArgumentNullException(nameof(text)); 
15    }
16
17    
18}

然后我们希望HtmlElement可以优雅的转换为字符串（应包含缩进）

 1    // 在 HtmlElement 类中
 2    private const int IndentSize = 2; // 定义缩进为两个空格
 3
 4
 5    private string ToStringImpl(int indent)
 6    {
 7        var sb = new StringBuilder();
 8        var i = new string(' ', indent * IndentSize);
 9        sb.AppendLine($"{i}<{Name}>");
10        if (!string.IsNullOrEmpty(Text))
11        {
12            sb.Append(new string(' ', (indent+1) * IndentSize));
13            sb.AppendLine(Text);
14        }
15
16        foreach (var e in Elements)
17        {
18            string s = e.ToStringImpl(indent + 1);
19            sb.Append(s);
20        }
21        sb.AppendLine($"{i}</{Name}>");
22        return sb.ToString();
23    } //使用递归DFS实现了元素的字符串输出
24
25    public override string ToString()
26    {
27        return ToStringImpl(0);
28    }

我们仍然不满意，还想要一个更简单的构建方式，于是我们自己写一个HtmlBuilder类

HtmlBuilder#

考虑HtmlBuilder应该以一个根元素为基础，可以添加子元素，可以清空子元素

 1public class HtmlBuilder
 2{
 3    private readonly string rootName; 
 4    private HtmlElement _root = new HtmlElement();
 5    
 6    public HtmlBuilder(string rootName)
 7    {
 8        this.rootName = rootName ?? throw new ArgumentNullException(nameof(rootName));
 9        _root.Name = rootName ?? throw new ArgumentNullException(nameof(rootName));
10    }
11
12    public void AddChild(string childName ,string childText)
13    {
14        var e = new HtmlElement(childName, childText);
15        _root.Elements.Add(e);
16    }
17
18    public override string ToString()
19    {
20        return _root.ToString();
21    }
22
23    public void Clear()
24    {
25        _root = new HtmlElement{Name = rootName};
26    }
27}

上述代码中专门保留rootName是因为我们希望Clear()后仍然保持其根元素

调用

 1public class Demo
 2{
 3    static void Main(string[] args)
 4    {
 5        var builder = new HtmlBuilder("ul");
 6        builder.AddChild("li","hello");
 7        builder.AddChild("li","world");
 8        Console.WriteLine(builder.ToString());
 9    }
10}

我们就得到了较为美观的输出

1
2<ul>
3  <li>
4    hello
5  </li>
6  <li>
7    world
8  </li>
9</ul>

流构建器#

注意到StringBuilder可以使用一种Fluent Interface Pattern的流调用方法 如

1sb.Append("a").Append("b").Append("c");

我们希望我们的HtmlBuilder也具有这样的功能，只需要

1public HtmlBuilder AddChild(string childName ,string childText)
2{
3    var e = new HtmlElement(childName, childText);
4    _root.Elements.Add(e);
5    return this;
6}//将原来的void返回值改为返回对自身的引用即可

于是我们可以这样调用

1builder.AddChild("li","hello").AddChild("li","world");

流式构建器的继承#

问题#

对普通构建者的继承不会引起什么问题，而当使用流式构建者时，问题则棘手起来。

先来看一个简单的例子，我们有Person类，它具有名字和职位两个属性。同时有一个流构建者只处理名字。

 1public class Person
 2{
 3    public string Name;
 4    public string Position;
 5    public override string ToString()
 6    {
 7        return $"{nameof(Name)}: {Name}, {nameof(Position)}: {Position}";
 8    }
 9}
10public class PersonInfoBuilder
11{
12    protected Person person = new Person();
13    //这里是protected因为我们一会将处理继承关系
14
15    public PersonInfoBuilder Called(string name)
16    {
17        person.Name = name;
18        return this;
19    }
20}

现在我们有新的业务需求，即现在需要Builder能同时处理职位的构建，我们遵循开闭原则使用新类继承PersonInfoBuilder

1public class PersonJobBuilder: PersonInfoBuilder
2{
3    public PersonJobBuilder WorksAsA(string position)
4    {
5        person.Position = position;
6        return this;
7    }
8}

现在我们尝试调用PersonJobBuilder

1// bad code
2PersonJobBuilder jb = new PersonJobBuilder();
3jb.Called("Li Bad").WorksAsA("Manager");

注意到这段代码是报错的，原因是Called方法返回的是一个PersonInfoBuilder，而PersonInfoBuilder不能处理职位相关事项，不具备WorksAsA方法

问题本质： 当 Fluent 方法需要返回 this 时，在继承链中，父类的方法如果返回的是父类类型 (PersonInfoBuilder)，那么子类的方法链就会在调用父类方法后中断，无法继续调用子类的方法。

解决方案#

使用递归泛型

首先我们来创建一个PersonBuilder的抽象类，将Person的存储和Build方法提取到抽象类里。

1public abstract class PersonBuilder
2{
3    protected Person person = new Person();
4
5    public Person Build()
6    {
7        return person;
8    }
9}

然后用递归泛型改写PersonInfoBuilder

 1public class PersonInfoBuilder<TSelf> : PersonBuilder
 2where TSelf : PersonInfoBuilder<TSelf>
 3{
 4
 5    public TSelf Called(string name)
 6    {
 7        person.Name = name;
 8        return (TSelf)this;
 9    }
10}

理解： 给PersonInfoBuilder添加了一个泛型TSelf，TSelf用来存放子类的类型，于是Called方法就可以通过类型转换return子类的类型，为了保证TSelf一定是子类的类型，我们用where语句限制TSelf继承于该类。

我们可能下意识这样使用

1// bad code
2public class PersonJobBuilder: PersonInfoBuilder<PersonJobBuilder>
3{
4    public PersonJobBuilder WorksAsA(string position)
5    {
6        person.Position = position;
7        return this;
8    }
9}

但是假想如果有类又继承PersonJobBuilder的话，由于PersonJobBuilder被固定，继承PersonJobBuilder的类又将不能正确工作。因此这决不是一个好注意。

正确的写法应该是在PersonJobBuilder上继续泛型

1public class PersonJobBuilder<TSelf> : PersonInfoBuilder<PersonJobBuilder<TSelf>>
2    where TSelf : PersonJobBuilder<TSelf>
3{
4    public TSelf WorksAsA(string position)
5    {
6        person.Position = position;
7        return (TSelf)this;
8    }
9}

理解： PersonJobBuilder也具有类型TSelf的泛型，也用where限制TSelf继承于该类，同时PersonJobBuilder中的泛型从PersonJobBuilder改为PersonJobBuilder，仅此而已

当我们高兴的尝试使用PersonJobBuilder时，我们发现PersonJobBuilder并不能直接构建。这样的泛型类都不能直接使用，必须用一个类继承泛型类，才能使用。

所以我们在Person中写一个Builder类，它继承于PersonJobBuilder<Person.Builder>，同时给Person类添加相应的构建方法

1// 在Person类内
2public class Builder:PersonJobBuilder<Builder>
3{
4}
5
6public static Builder New => new Builder();

于是我们现在可以调用

1var person = Person.New.
2    Called("Wang").
3    WorksAsA("LaoBan").
4    Build();
5Console.WriteLine(person);

得到输出

Name: Wang, Position: LaoBan

分步构建器#

设想这样一个场景，我们现在要做车的创建者，车子具有类型，小轿车(Sedan)或跨界车(CrossOver)，小轿车的尺寸必须在15到17之间，而跨界车必须在17-20之间，如果超出范围就要报错。

我们注意到该构建过程具有明显的分步性，必须先知道类型，才能判断尺寸，这就引出了分步构建者

先定义车类

 1public enum CarType
 2{
 3    Sedan,
 4    Crossover
 5}
 6
 7public class Car
 8{
 9    public CarType type;
10    public int WheelSize;
11}

应用接口隔离原则我们将车的构建过程分成几个独立的接口

 1public interface ISpecifyCarType
 2{
 3    ISpecifyWheelSize OfType(CarType type);
 4}
 5public interface ISpecifyWheelSize
 6{
 7    IBuildCar WithWheels(int size);
 8}
 9    public interface IBuildCar
10{
11    public Car Build();
12}

其中ISpecifyCarType用来指定车的类型，它随后返回一个ISpecifyWheelSize接口进一步指定车的轮子尺寸，再返回IBuildCar进一步构建车对象

在CarBuilder类中，我们将构建过程的实现封装到Impl里

 1public class CarBuilder
 2{
 3    private class Impl:ISpecifyWheelSize,ISpecifyCarType,IBuildCar
 4    {
 5        private Car _car = new Car();
 6        public ISpecifyWheelSize OfType(CarType type)
 7        {
 8            _car.type = type;
 9            // work as ISpecifyCarType
10            return this;
11            // this is ISpecifyWheelSize
12        }
13        public IBuildCar WithWheels(int size)
14        {
15            switch (_car.type)
16            {
17                case CarType.Crossover when size is < 17 or > 20:
18                case CarType.Sedan when size is < 15 or > 17:
19                    throw new ArgumentException("Car size out of range");
20                default:
21                    break;
22            }
23            _car.WheelSize = size;
24            // work as ISpecifyWheelSize
25            return this;
26            // this is IBuildCar
27        }
28
29
30        public Car Build()
31        {
32            return _car;
33        }
34    }   
35    public static ISpecifyCarType Create()
36    {
37        return new Impl();
38    }
39}

注意到上述Impl类同时实现了ISpecifyWheelSize,ISpecifyCarType,IBuildCar三个接口，所以它只需要不停return this就可以让构建过程不断递进。

使用如下

1var car = CarBuilder.Create().OfType(CarType.Sedan).WithWheels(20).Build();

函数式构建器#

我们可以使用函数式的方式来构造一个PersonBuilder类，我们让它存储下我们想要对对象进行的所有操作的列表，然后在Build时再对初始对象进行所有这些操作并返回

 1public sealed class PersonBuilder
 2{
 3    private List<Func<Person, Person>> actions =  new List<Func<Person, Person>>();
 4
 5    private PersonBuilder AddAction(Action<Person> action)
 6    {
 7        actions.Add(p=>
 8        {
 9            action(p);
10            return p;
11        });
12        return this;
13    }
14
15    public Person Build() => actions.Aggregate(new Person(), (acc, action) => action(acc)); 
16    public PersonBuilder Called(string name) => Do(p => p.Name = name);
17    public PersonBuilder Do(Action<Person> action)=>AddAction(action);
18    
19}

我们使用sealed关键字密封该类，强调我们不需要继承就能实现想要的功能。

那么我们如何不通过继承来添加新的功能呢？

答案是通过C#的静态拓展方法

1public static class PersonBuilderExtensions
2{
3    public static PersonBuilder WorksAsA(this PersonBuilder builder, string position)
4        => builder.Do(p => p.Position = position);
5}

此时我们可以直接使用WorksAsA

1var person = new PersonBuilder().Called("Wang").WorksAsA("LaoBan").Build();

注意到，函数式创建者的工作方式其实都是类似的，所以我们直接将函数式构建者写为泛型，然后再特化使用

 1public abstract class FunctionBuilder<TSubject, TSelf>
 2where TSelf : FunctionBuilder<TSubject, TSelf>
 3where TSubject: new()
 4{
 5    private List<Func<TSubject, TSubject>> actions =  new List<Func<TSubject, TSubject>>();
 6    
 7    private TSelf AddAction(Action<TSubject> action)
 8    {
 9        actions.Add(p=>
10        {
11            action(p);
12            return p;
13        });
14        return (TSelf)this;
15    }
16    
17    public TSubject Build() => actions.Aggregate(new TSubject(), (acc, action) => action(acc)); 
18    public TSelf Do(Action<TSubject> action)=>AddAction(action);
19    
20}
21
22public sealed class PersonBuilder:FunctionBuilder<Person,PersonBuilder>
23{
24    public PersonBuilder Called(string name)=> Do(p => p.Name = name);
25}
26
27public static class PersonBuilderExtensions
28{
29    public static PersonBuilder WorksAsA(this PersonBuilder builder, string position)
30        => builder.Do(p => p.Position = position);
31}
32

分派构建器#

我们可以将一个对象的属性分拆成不同的方面，交给不同的构建器负责，最后再封装给一个构建器

例子#

比如我们有一个Person类，它具有StreetAddress,Postcode,City等居住地相关属性和CompanyName,Position,AnnualIncome等工作相关属性

 1public class Person
 2{
 3    public string? StreetAddress,Postcode,City;
 4    public string? CompanyName, Position;
 5    public int AnnualIncome;
 6    public override string ToString()
 7    {
 8        return $"{nameof(StreetAddress)}: {StreetAddress}, {nameof(Postcode)}: {Postcode}, {nameof(City)}: {City}, {nameof(CompanyName)}: {CompanyName},  {nameof(Position)}: {Position}, {nameof(AnnualIncome)}: {AnnualIncome}";
 9    }
10}

然后再写一个PersonBuilder类（该类是一个外壳，并不处理具体构建过程）

1public class PersonBuilder // facade
2{
3    protected Person Person = new Person();
4}

然后我们写PersonAddressBuilder和PersonJobBuilder两个类继承于PersonBuilder，分别处理地址和工作相关的属性的构建

 1public class PersonAddressBuilder : PersonBuilder
 2{
 3    public PersonAddressBuilder(Person person)
 4    {
 5        Person = person;
 6    }
 7
 8    public PersonAddressBuilder At(string streetAddress)
 9    {
10        Person.StreetAddress = streetAddress;
11        return this;
12    }
13
14    public PersonAddressBuilder WithPostcode(string postcode)
15    {
16        Person.Postcode = postcode;
17        return this;
18    }
19
20    public PersonAddressBuilder In(string city)
21    {
22        Person.City = city;
23        return this;
24    }
25}
26public class PersonJobBuilder:PersonBuilder
27{
28    public PersonJobBuilder(Person person)
29    {
30        this.Person = person;
31    }
32
33    public PersonJobBuilder At(string companyName)
34    {
35        Person.CompanyName = companyName;
36        return this;
37    }
38
39    public PersonJobBuilder AsA(string position)
40    {
41        Person.Position = position;
42        return this;
43    }
44    public PersonJobBuilder Earning(int amount)
45    {
46        Person.AnnualIncome = amount;
47        return this;
48    }
49}

将PersonJobBuilder和PersonAddressBuilder的对象作为PersonBuilder的公共属性

1public class PersonBuilder // facade
2{
3    protected Person Person = new Person();
4    public PersonJobBuilder Works => new PersonJobBuilder(Person); 
5    public PersonAddressBuilder Lives => new PersonAddressBuilder(Person);
6
7    public static implicit operator Person(PersonBuilder b) => b.Person;
8}

上面代码中的operator重载了PersonBuilder向Person的隐式转换，使得PersonBuilder构建对象使用时更方便

于是我们可以这样使用构建器

1var pb = new PersonBuilder();
2Person person = pb.Works.At("Huawei").AsA("Manager").Earning(1000000)
3    .Lives.At("Nanluoguxiang").In("Beijing").WithPostcode("666");

总结#

  1. 构建器是用来构建对象的独立的组件
  2. 你可以通过构造函数生成构建器，也可以通过静态函数返回构建器
  3. 如果要让构建器实现流式接口，只需要return this，如果还要实现继承，使用递归泛型
  4. 一个对象的不同方面可以由不同的构建器通过一个基类并联起来
